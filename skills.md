# Skills — lessons from bugs already hit in this codebase

This file exists so future work doesn't re-discover the same root causes from
scratch. Each entry: the **symptom** you'd actually see, the **root cause**,
and the **fix pattern** to apply next time something looks similar.

---

## 1. "Screen A did something, but screen B doesn't know" → plain hooks don't share state

**Symptom:** One part of the UI updates correctly, but a different part of
the UI showing related data doesn't — until the app is restarted or that
other part happens to refetch on its own. Examples already hit:
- Password reset: `reset-password.tsx` cleared `isPasswordRecovery`, but the
  root layout (which decides where to route you) never found out, so the
  user got stuck on the success screen forever.
- Friends tab badge: resolving a challenge/request in `friends.tsx` never
  updated the badge count in `app/(tabs)/_layout.tsx`.
- `StickerDetailView`'s "Challenge" button silently always saw an empty
  friends list, because nobody ever passed it a `currentUserId` prop.

**Root cause:** A React hook (`useAuth()`, `useFriends()`, `useChallenges()`)
is just a function. Calling it from two different components does **not**
give them a shared variable — each call site gets its **own independent**
`useState`/`useEffect`, with its own private copy of the data and its own
fetch. They look the same at first (same query, same initial result) but
diverge the moment one of them mutates and the other doesn't hear about it.

**Fix pattern:** If more than one component needs to *agree* on a piece of
state (auth session, friends list, challenge inbox), it must live in a
single shared instance via React Context, not be re-created per call site:

```ts
function useThingState() { /* the actual useState/useEffect logic */ }
const ThingContext = createContext<ReturnType<typeof useThingState> | null>(null);
export function ThingProvider({ children }) {
  return <ThingContext.Provider value={useThingState()}>{children}</ThingContext.Provider>;
}
export function useThing() {
  const ctx = useContext(ThingContext);
  if (!ctx) throw new Error('useThing must be used within a ThingProvider');
  return ctx;
}
```
Mount the Provider once, near the root (`app/_layout.tsx`), nesting
`AuthProvider → FriendsProvider → ChallengesProvider` since the latter two
read `useAuth()` internally to get the current user ID (no more threading a
`userId`/`currentUserId` prop through every call site — which is itself a
prop that's easy to forget to pass, as `StickerDetailView` proved).

**When this does NOT apply:** purely local, single-screen UI state (form
inputs, modal open/closed, animation values) — that's correctly a plain
`useState` in the component that owns it. Context is only for state that
*multiple independent components* need to read the same live value of.

**Where this lives in this codebase:** `hooks/useAuth.ts`, `hooks/useFriends.ts`,
`hooks/useChallenges.ts` are all Context-based now. If you add a new piece of
cross-screen state, follow the same pattern rather than a bare hook.

## 2. "The image/file never loads for [some other user]" → Storage RLS is per-owner-folder

**Symptom:** A signed URL silently never resolves (`data` comes back `null`,
no thrown error) when generated by anyone other than the file's original
uploader. UI shows a permanent placeholder or spinner with no error message.

**Root cause:** `supabase/migrations/001_initial_schema.sql` documents (but
does **not** create via SQL — it's a manual dashboard policy, so it won't
show up if you grep migrations for it) a storage policy on `sticker-images`:

```sql
USING (bucket_id = 'sticker-images' AND auth.uid()::text = (storage.foldername(name))[1])
```

i.e. every file lives at `{ownerUserId}/{filename}`, and **only that owner**
can read/write/delete it. This is the standard, correct pattern for private
per-user files — but it means **any feature where one user needs to see
another user's file** (challenge snapshots, future "view a friend's
collection", anything cross-user) will silently fail if you just call
`createSignedUrl` from the client as the *other* user. Minting a signed URL
is itself a read against `storage.objects`, so it's gated by this same rule
— a signed URL does not bypass RLS, it only changes what you get *if* RLS
allows the request.

**Fix pattern — two options, pick based on whether the file should become
permanently owned by the other user or just viewable:**
- **Permanent transfer of ownership** (e.g. a sticker the receiver just won):
  have a service-role edge function `.copy()` the file into the new owner's
  own folder, and point the DB row's path at the copy. See
  `submit-challenge-answer/index.ts`'s win branch. After this, completely
  normal client-side `createSignedUrl` calls work for that user forever —
  no special-casing needed anywhere else in the app.
- **Temporary cross-user viewing, no ownership change** (e.g. viewing a
  challenge snapshot before you've won it): write a small edge function that
  (a) verifies the caller is actually authorized to see this specific file
  via your own application logic (not RLS), then (b) uses the
  `SUPABASE_SERVICE_ROLE_KEY` client (which bypasses RLS entirely) to mint
  the signed URL and return just the URL string. See
  `get-challenge-image/index.ts`. Never expose the service-role key to the
  client — it only ever lives inside edge functions.

**Generalize this:** any time a new feature lets User A see/use a resource
that's stored under User B's folder, ask up front: "transfer a copy, or read
through a verified server-side check?" — don't let it surface later as a
silently-broken image.

## 3. `supabase.functions.invoke()` errors are useless by default

**Symptom:** An edge function clearly returns a helpful JSON body like
`{ "error": "A pending challenge with this sticker already exists" }`, but
the app shows the user (or you, debugging) only:
`"Edge Function returned a non-2xx status code"`.

**Root cause:** the JS client wraps any non-2xx response in a generic
`FunctionsHttpError` and does **not** automatically parse the body for you.
The real response is still there, on `error.context` (a `Response` object)
— it just requires an extra `await error.context.json()` to get at.

**Fix pattern:** never surface `res.error.message` directly to a user.
Always unwrap it first — see `lib/functionError.ts`:
```ts
export async function getFunctionErrorMessage(error: unknown): Promise<string> {
  if (error instanceof FunctionsHttpError) {
    try {
      const body = await error.context.json();
      if (typeof body?.error === 'string') return body.error;
    } catch {}
  }
  return error instanceof Error ? error.message : 'Something went wrong';
}
```
Apply this in every hook function that calls `supabase.functions.invoke(...)`
before returning/throwing the error to UI code. If a specific known error
string deserves friendlier copy than the backend's literal message (e.g.
"already exists" → "wait for your friend to finish it"), remap it once at
this same boundary, not scattered across every screen that calls the hook.

## 4. This project's build/test loop has no fast refresh

**Context:** this is a managed Expo project (no `ios`/`android` native
folders) using **EAS Build**, not Expo Go. There is no live Metro dev-client
connection in the normal test flow — every JS/TS change requires a full
`eas build` and a fresh install on the device to test manually.

**Practical commands:**
```bash
npx eas-cli build --platform ios --profile preview --non-interactive
# … wait, returns an expo.dev/.../builds/<id> install link
```
Run it with `run_in_background: true` (it takes several minutes) and report
the install link back once it completes — don't poll, you'll get a
completion notification. There is currently no Android build profile in use
(`eas build:list --platform android` returns empty) — this has only been
tested on iOS so far.

**Don't forget:** if a fix touches a Supabase **edge function**, that's a
*separate* deploy from the app build — `npx supabase functions deploy <name>`
(or list multiple names) BEFORE kicking off the EAS build, since the app
just calls the function by name at runtime and won't see source changes
otherwise. Match the existing functions' convention: deploy without
`--no-verify-jwt` (default JWT verification) — every function in this
project does its own `userClient.auth.getUser()` check anyway, so the
platform-level check is just an extra layer, not a behavior change.

## 5. Read AGENTS.md's Expo version pointer before assuming any Expo API

This project pins to **Expo SDK 54** specifically — `AGENTS.md` calls out
that Expo has changed enough between versions that training-data assumptions
about Expo APIs are unreliable. Check
`https://docs.expo.dev/versions/v54.0.0/` for the actual current shape of
any Expo module API before using it, especially anything storage/media/
notifications-related — these are exactly the modules that have shifted
behavior across recent SDKs (e.g. `expo-notifications` remote push no longer
works in Expo Go as of SDK 53+, part of why this project is on EAS dev/preview
builds rather than Expo Go in the first place).
